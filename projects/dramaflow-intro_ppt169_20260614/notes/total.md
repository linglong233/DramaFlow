# 01_封面

大家好，今天我来介绍 DramaFlow —— 一个由 AI 驱动的短剧全流程生产平台。它是一个 TypeScript 单仓多包项目，由前端、后端、轮询 worker 和一个共享契约包四部分组成，覆盖了从剧本创作到成片导出的完整链路。先来看看，它究竟想解决什么问题。

---

# 02_解决的问题

传统短剧生产有几道明显的痛点。第一，工具链割裂——剧本、分镜、绘图、视频、配音、剪辑散落在不同工具里，素材全靠手动搬运。第二，AI 能力难以串成闭环，文本、图片、视频、TTS 各自为政，产出物没法被下游复用。第三，协作和审阅缺位，版本、评论、审计和权限依赖文件名和聊天记录。第四，成片组装的最后一公里——拿到产出物后，仍要手工拼时间线、对齐字幕再导出。第五，角色和场景的视觉一致性难维持，反复重生成时外观容易漂移。DramaFlow 的回答，就是把这五道关卡收敛到一个平台，形成一条 AI 全流程闭环。

---

# 03_项目定位

带着这些痛点，来看 DramaFlow 的定位。它面向的是导演与制作团队，核心目标是把短剧生产做成一条 AI 全流程闭环。简单说，就是让一个创作者从一句话梗概出发，在同一平台内完成剧本、分镜、图片、视频、配音、时间线组装直到导出。这一页用三根支柱概括了它覆盖的范围：创作、媒体生产，以及协作与导出。

---

# 04_核心能力

这一页展开六大核心能力。AI 写作支持剧本、梗概、分镜和改写，并以流式方式生成，还支持对话式问答。分镜工作台提供拖拽重排、多选和镜头三栏详情。图片和视频生成都支持逐镜头和批量两种模式，并能在多个 provider 之间灵活切换。配音可以逐镜头或按场景批量合成。最后，协作审阅把版本生命周期、线程评论、审计和权限统一管理起来。

---

# 05_技术栈

技术栈上，DramaFlow 是全栈 TypeScript。前端用 Next.js 十五和 React 十九，基于 App Router，样式是纯 CSS 语义化，国际化也是自研的。后端是 NestJS 十一，认证用 JWT 配合 argon2，实时推送走 Socket.IO 网关。数据层是 PostgreSQL 加 Prisma，一共二十六个模型。基础设施方面用 npm workspaces 管理多包，存储支持本地和 S3 两种后端，导出依赖 FFmpeg，整体可以通过 Docker Compose 一键拉起。

---

# 06_架构总览

架构上是一个清晰的四层 monorepo。最上层是 apps/web，Next.js 前端，用一个统一的工作区 shell 通过 URL 的 mode 参数切换四种模式。往下是 apps/api，NestJS 服务，承载了几乎全部业务逻辑和 provider 编排。第三层是 apps/worker，只有大约一百行的轮询循环，负责认领、执行和重试任务，本身不含业务逻辑。底层贯穿四件套的是 packages/shared，它是跨栈领域类型和业务规则的唯一真相源。底部这条深色横带，则是贯穿各层的横切关注点：数据库、实时通信、存储和导出。

---

# 07_AI生成能力

这一页把 AI 生成能力串成一条端到端的生产链。第一段是文本生成，产出剧本和梗概；然后是图片生成，对接 Gemini、SD WebUI 和 ComfyUI，产出镜头图；接着视频生成产出镜头视频；再往后是 TTS 配音，产出角色语音；最后由合成导出把时间线组装成成片。关键点是，每个阶段的产出物都能被下游复用，形成一条真正闭环的流水线。

---

# 08_创作工作流

文档在生产过程中遵循一套明确的审阅生命周期：从草稿，到已提交，到待审，最终走向通过或驳回。驳回后会回到草稿重新修改，形成一个回环。所有审阅与审批的业务规则都集中在共享包的 business-rules 文件里，保证前后端和 worker 行为一致。除了状态流转，版本管理还支持采纳、还原和差异对比，而剧本与世界观角色之间则通过 WebSocket 做实时双向同步。

---

# 09_工作区与协作

整个创作过程发生在一个统一的工作区里，通过 URL 的 mode 参数在四种模式间切换。信息模式展示项目面板和成员；文档模式是主战场，涵盖查看、编辑、生成和版本，世界观和媒体也都映射到这里；任务模式跟踪生成任务和批量作业；时间线模式负责时间线编辑、自动组装和导出提交。一套 shell、四种模式，把分散的工具收敛到一个连续的工作流。

---

# 10_项目状态

最后看一下当前状态。DramaFlow 目前已经开发就绪，五项关键能力都已落地：PostgreSQL 加 Prisma 的持久化、轮询 worker 任务流、Socket.IO 实时更新、provider 对接加 mock 回退，以及 FFmpeg 视频导出。面向未来，主要规划了两件事：一是引入 Redis 或 BullMQ 来替换当前的轮询 worker，提升吞吐和可靠性；二是完成生产化加固，补齐可观测性、限流和运维能力。

---

# 11_结尾

这就是 DramaFlow 的全貌 —— 一个让短剧创作如行云流水的 AI 全流程平台。它现在开发就绪，欢迎大家一起使用、贡献和反馈。运行起来很简单：Node.js 二十四以上，装好依赖，跑一次构建，就能在本地三百端口打开前端、四百端口访问后端。感谢大家的观看。
