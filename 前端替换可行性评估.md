# 前端替换可行性评估

## 结论

- 在当前 `DramaFlow` 架构下，**替换前端是可行的**。
- 如果目标是“保留现有 `apps/api`、`apps/worker`、`packages/shared`，只更换 `apps/web`”，整体风险可控。
- 当前前端主要是一个直接消费 REST API 的客户端应用，并没有深度依赖 Next.js 的服务端特性。

## 当前架构判断

### 1. 前后端边界已经基本分离

- 前端集中通过 `apps/web/lib/api.ts` 调用后端。
- 后端公开接口集中在以下模块：
  - `apps/api/src/auth/auth.controller.ts`
  - `apps/api/src/workspace/workspace.controller.ts`
  - `apps/api/src/jobs/jobs.controller.ts`
  - `apps/api/src/storage/uploads.controller.ts`
  - `apps/api/src/admin/admin.controller.ts`
- `apps/api/src/main.ts` 已启用 CORS，并提供 Swagger 文档 `/docs`。

### 2. 当前前端主要是客户端壳

- `apps/web/app` 下的 page 文件普遍较薄，主要负责路由入口。
- 主要业务交互集中在客户端组件：
  - `apps/web/components/login-panel.tsx`
  - `apps/web/components/dashboard-overview.tsx`
  - `apps/web/components/project-workspace.tsx`
  - `apps/web/components/navigation-shell.tsx`
- 当前没有发现以下强耦合点：
  - Next API Routes
  - Server Actions
  - 自定义 BFF
  - 依赖 RSC 预取的核心业务链路

### 3. 鉴权方案对替换前端友好

- 当前登录态由前端保存在 `localStorage`。
- 请求通过 `Authorization: Bearer <token>` 调用后端。
- 这是一种标准浏览器前端集成方式，React SPA、Vue SPA、Nuxt、SvelteKit、移动端壳都可以复用。

## 当前需要注意的耦合点

### 1. 共享类型没有被前端充分复用

- `apps/web` 虽然依赖了 `@dramaflow/shared`，但实际页面里仍有较多本地手写接口类型。
- 典型位置：
  - `apps/web/components/dashboard-overview.tsx`
  - `apps/web/components/project-workspace.tsx`
- 这不影响“能不能换前端”，但会影响迁移质量：
  - 如果继续手写类型，迁移速度快但长期维护成本高。
  - 如果顺手把接口契约统一到 `packages/shared`，迁移质量会更高。

### 2. 当前登录态更偏客户端方案

- 现在的方案更适合 SPA。
- 如果未来改成重 SSR 的前端框架，通常会考虑把 `localStorage` token 模式升级为 cookie/session 模式。
- 这不是替换前端的前置条件，但属于后续可选优化。

### 3. 上传链路需要按接口协议复刻

- 上传不是单纯的前端表单提交。
- 需要按后端现有方式调用：
  1. `POST /uploads` 获取上传目标
  2. `PUT /uploads/direct/:key` 直传内容
  3. `GET /assets/:id/url` 读取资源地址
- 如果新前端支持浏览器 `fetch` / `XMLHttpRequest`，这条链路可直接迁移。

## 推荐替换策略

### 最低风险方案

- 保留：
  - `apps/api`
  - `apps/worker`
  - `packages/shared`
- 新建一个新的前端应用替代 `apps/web`。
- 第一阶段只复刻现有能力：
  - 登录页
  - 工作台
  - 项目工作区
  - 平台后台
  - 团队后台
- 第二阶段再做体验升级：
  - 更好的状态管理
  - 更清晰的类型契约
  - 更成熟的任务轮询和错误处理

### 技术选择判断

- React + Vite：最顺，迁移成本最低。
- Vue / Nuxt：可行，主要成本在页面与状态层重写。
- SvelteKit：可行，适合想要更轻运行时的场景。
- 移动端 / 小程序前端：也可行，但需要单独适配上传与 token 刷新策略。

## 迁移时的验证清单

- 注册、登录、退出是否正常。
- Bearer Token 是否正确注入。
- `401` 和 token 失效是否有统一处理。
- 团队和项目列表是否正常拉取。
- 项目创建是否正常。
- 项目页是否能查看文档、版本、评论。
- 剧本、分镜、生图、生视频任务是否能正常提交。
- `GET /jobs/:id` 的轮询状态是否正常。
- 上传接口 `POST /uploads` + `PUT /uploads/direct/:key` 是否正常。
- 平台后台和团队后台概览是否正常展示。

## 建议的迁移原则

- 把 Nest API 继续视为唯一真相源，不在新前端额外复制业务规则。
- 如果接口结构需要调整，应优先更新 `packages/shared` 和后端，再更新新前端。
- 如果只是替换视图层，不建议同步改动后端鉴权、任务系统、存储系统和数据层。

## 总结

- 当前仓库已经具备“后端独立、前端可替换”的基本条件。
- 真正的成本不在后端接口层，而在以下两点：
  - 页面和交互重建
  - 前端接口类型从“分散手写”整理为“稳定契约”
- 因此，如果你只是觉得 Next.js 不顺手，而不是要推翻整套系统，最合理的路线就是：**保留现有后端与 worker，单独替换 `apps/web`。**
